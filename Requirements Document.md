# Obsidian Librarian — Requirements Document

**Version:** 0.1 (Draft)
**Date:** 2026-06-02
**Status:** In Design

---

## 1. Project Overview

The Obsidian Librarian is a locally-running, AI-powered knowledge management system. It autonomously organizes an Obsidian vault and exposes it as a RAG (Retrieval-Augmented Generation) backend for home AI infrastructure. The user should never need to manually organize notes — the system handles structure, tagging, linking, deduplication, and enrichment automatically.

---

## 2. User Context & Constraints

- **User:** Single user, AuDHD, prefers zero manual maintenance overhead
- **Hardware:** Apple M5 Max, 48GB unified RAM, 2TB disk, macOS
- **LLM Runtime:** LM Studio (local, OpenAI-compatible API)
- **Primary notes app:** Obsidian (existing vault with ~2,300+ notes)
- **Privacy requirement:** All processing must be 100% local — no cloud, no external APIs
- **Autonomy requirement:** System must make decisions and act without requiring user input for routine operations

---

## 3. Functional Requirements

### 3.1 Vault Organization

**FR-01 — Folder Structure Enforcement**
The system shall maintain a consistent folder structure within the vault. When notes are found outside their correct location, the system shall move them automatically according to the active folder schema.

**FR-02 — AI-Determined Structure**
The folder schema shall be determined and evolved by the AI based on observed content patterns. The user shall not be required to define or maintain the folder schema. The schema shall be documented in a human-readable config file that the user can override.

**FR-03 — Naming Convention Enforcement**
The system shall detect and optionally correct note filenames that violate detected naming conventions (e.g., inconsistent date formats, special characters, excessively long names).

**FR-04 — Automatic Note Moving**
When a note is created or modified, the system shall evaluate its content and move it to the appropriate folder if it is misplaced. Moves shall be logged with rationale.

**FR-05 — Daily Note Consolidation**
Notes matching the daily note pattern (`Daily - MM-DD-YYYY.md`) found at vault root shall be automatically moved to a `Daily Notes/` folder.

**FR-06 — Archive Management**
Notes that have not been accessed or modified in more than 180 days and have no incoming links shall be flagged and optionally moved to an `Archive/` folder. Threshold shall be configurable.

---

### 3.2 Note Enrichment

**FR-07 — Frontmatter Injection**
Notes without YAML frontmatter shall have it added automatically. Minimum frontmatter fields:
```yaml
tags: []
topic: ""
note_type: ""          # reference | fleeting | literature | permanent | daily | project
status: ""             # draft | active | archived
privacy_tier: 0        # 0=public, 1=personal, 2=sensitive, 3=private
created: ""
modified: ""
```

**FR-08 — Auto-Tagging**
The system shall suggest and apply tags based on note content. Tags shall be consistent with the existing vault taxonomy. The system shall not invent new tags without a minimum confidence threshold.

**FR-09 — Topic Classification**
Each note shall be assigned a primary `topic` field drawn from a taxonomy the system derives from the vault (e.g., `AI`, `Parenting`, `Computer Science`, `Health`).

**FR-10 — Note Type Classification**
Each note shall be classified by type: `reference`, `fleeting`, `literature`, `permanent`, `daily`, `project`, or `resource`.

**FR-11 — Automatic Backlinking**
When the system detects that note A references concepts covered in note B, it shall suggest or create a wikilink from A to B. User-defined link creation shall require explicit approval in non-autonomous mode.

---

### 3.3 Deduplication

**FR-12 — Exact Duplicate Detection**
Files with identical or near-identical content (>95% similarity) shall be flagged as duplicates. Backup files (matching pattern `_backup_YYYYMMDD_HHMMSS`) shall be automatically flagged for deletion.

**FR-13 — Semantic Duplicate Detection**
Notes covering substantially the same topic with high semantic similarity shall be surfaced for potential merging. Merging shall never be automatic — it shall require user approval.

**FR-14 — Backup File Cleanup**
Files generated by other tools as backups (identifiable by naming pattern) shall be offered for deletion with a preview of what will be removed.

---

### 3.4 Orphan & Stale Note Management

**FR-15 — Orphan Detection**
Notes with no incoming or outgoing wikilinks and no folder context indicating intentional isolation shall be flagged as orphans and surfaced for filing or deletion.

**FR-16 — Stale Note Detection**
Notes not modified in more than 90 days (configurable) shall be flagged as potentially stale and included in the health report. No automatic action shall be taken on stale notes.

---

### 3.5 Rule Registry & Learning

**FR-17 — Rule Registry**
The system shall maintain a persistent `rules_registry.yaml` file containing all learned and manually authored rules. Each rule shall include: ID, name, type, pattern, action, confidence score, hit count, creation date, and description.

**FR-18 — Rule-First Execution**
On every event, the system shall run all rules in the registry before invoking the LLM. If a rule matches, the LLM shall not be called. This is the primary mechanism for reducing token usage over time.

**FR-19 — Automatic Rule Generation**
After every 25 LLM decisions (configurable), the RuleWriter agent shall analyze recent decisions for generalizable patterns. When a pattern is detected with confidence ≥ 0.85, a new rule shall be written to the registry.

**FR-20 — Rule Types**
The system shall support four rule types:
- `regex` — filename or path pattern
- `frontmatter_pattern` — match on YAML fields
- `content_pattern` — regex on note body
- `python_callable` — arbitrary Python function for complex logic

**FR-21 — Rule Audit Trail**
Every rule execution shall be logged (rule ID, file affected, action taken, timestamp). Rules with zero hits after 90 days shall be flagged for review.

**FR-22 — Manual Rule Authoring**
The user shall be able to add rules manually by editing `rules_registry.yaml` or via the Web UI. The system shall validate rule syntax before accepting new rules.

---

### 3.6 Embeddings & RAG

**FR-23 — Continuous Embedding**
All notes shall be embedded using `nomic-embed-text`. New and modified notes shall be re-embedded within 60 seconds of a file save event (subject to debounce).

**FR-24 — Heading-Aware Chunking**
Notes shall be chunked at H2/H3 heading boundaries. Each chunk shall include metadata: source file path, heading title, chunk index, character range, and modification timestamp.

**FR-25 — Hybrid Search**
The RAG retrieval layer shall support hybrid search combining semantic vector similarity (ChromaDB cosine) and keyword search (BM25). The blend weight shall be configurable.

**FR-26 — Privacy-Filtered Retrieval**
All RAG queries shall respect the `privacy_tier` field on notes. External API queries shall default to returning only `tier 0` and `tier 1` content. Tier filtering shall be enforceable per query.

**FR-27 — Source Citations**
All RAG responses shall include citations in Obsidian wikilink format pointing to the source notes.

**FR-28 — External RAG API**
The system shall expose an OpenAI-compatible `/v1/chat/completions` endpoint accessible on the local network so that other AI tools (Home Assistant, other local LLMs, scripts) can query the vault.

---

### 3.7 Ingest Pipeline

**FR-29 — Real-Time File Watching**
The system shall use FSEvents (via `watchdog`) to monitor the vault in real time. File create, modify, and delete events shall be processed within 5 seconds (subject to 5-second debounce).

**FR-30 — Scheduled Full Scan**
A full vault re-scan shall run nightly at 2 AM via `launchd`. This scan shall re-embed notes with stale embeddings, regenerate the health report, and trigger RuleWriter review.

**FR-31 — New File Handling**
On new file creation: parse → embed → rule check → agent enrichment. Default behavior is fully automatic in autonomous mode.

**FR-32 — Delete Handling**
On file deletion: remove from ChromaDB, remove from librarian.db, log the deletion. No recovery action shall be taken automatically.

---

### 3.8 GUI — Menubar App

**FR-33 — Menubar Status**
The macOS menubar icon shall display current system status (healthy / warning / error) and show a summary dropdown including: note count, last run time, and pending action count.

**FR-34 — Quick Actions**
The menubar shall provide one-click access to: Run Now, Dry Run (preview mode), Pause/Resume, and Open Web Dashboard.

**FR-35 — Notification Support**
The system shall send macOS notifications for: completion of a run, critical errors, and when pending actions exceed a configurable threshold (default: 10).

---

### 3.9 GUI — Web Dashboard

**FR-36 — Vault Health Overview**
Dashboard home page shall show: total note count, orphan count, untagged note count, duplicate count, stale note count, last run timestamp, and rules registry size.

**FR-37 — Pending Actions Queue**
The dashboard shall display all pending actions (moves, renames, tag updates) with individual approve/reject controls and a bulk approve option.

**FR-38 — Rule Registry Browser**
The dashboard shall display all rules with hit count, confidence, creation date, and enable/disable toggle. Rules shall be editable inline.

**FR-39 — RAG Query Interface**
The dashboard shall include a chat interface that queries the vault via the RAG engine. Responses shall include source citations with links to notes.

**FR-40 — Knowledge Graph View**
The dashboard shall render a force-directed graph (D3.js) of note connections based on wikilinks and semantic similarity edges. Nodes shall be filterable by topic and note type.

**FR-41 — Audit Log Viewer**
The dashboard shall display a paginated, searchable log of all system actions with file, action, rule/agent, and timestamp columns.

---

### 3.10 CLI

**FR-42 — CLI Entry Point**
The system shall expose a `librarian` CLI command with the following subcommands:
- `librarian run` — trigger a full organization pass
- `librarian dry-run` — preview all proposed changes without applying them
- `librarian query "..."` — query the vault RAG from the terminal
- `librarian rules list` — display all rules
- `librarian rules add` — add a rule interactively
- `librarian status` — show vault health summary
- `librarian ingest <file>` — manually ingest a specific file
- `librarian serve` — start the API + web dashboard

---

### 3.11 Privacy & Security

**FR-43 — Local-Only Processing**
All LLM inference, embedding generation, and data storage shall occur on the local machine. No note content, metadata, or query shall be transmitted to any external server.

**FR-44 — Privacy Tier Enforcement**
Notes tagged with `privacy_tier: 2` (sensitive) or `privacy_tier: 3` (private) shall never be returned via the external API. They shall be excluded from RAG retrieval unless the query originates from localhost.

**FR-45 — Sensitive Category Detection**
The Classifier agent shall automatically detect notes containing sensitive content categories (medical, therapy, financial, legal, credentials) and assign `privacy_tier: 2` unless the user has explicitly set a lower tier.

**FR-46 — No Credential Storage**
The system shall not store LLM API keys (LM Studio is local and unauthenticated by default). If authentication is added, credentials shall be stored in macOS Keychain, not in config files.

## 4. Non-Functional Requirements

### 4.1 Performance

**NFR-01** — Single note ingest (parse + embed + rule check) shall complete in under 10 seconds on the target hardware.

**NFR-02** — RAG query response (embed query + vector search + LLM synthesis) shall complete in under 15 seconds for typical queries.

**NFR-03** — Full vault re-scan of 2,500 notes shall complete in under 30 minutes (nightly batch window).

**NFR-04** — The system shall consume no more than 4GB RAM when idle (watcher only, no active inference).

**NFR-05** — Rule registry execution (all rules, no LLM) shall add no more than 500ms to any file processing event.

### 4.2 Reliability

**NFR-06** — The FSEvents watcher shall automatically restart within 30 seconds of an unexpected crash.

**NFR-07** — ChromaDB writes shall be atomic. Partial writes on crash shall not corrupt the vector store.

**NFR-08** — All vault modifications (moves, renames, frontmatter writes) shall be logged to `run_log.jsonl` before execution, enabling rollback audit.

**NFR-09** — A dry-run mode shall be available for all destructive operations. No file shall be moved or deleted without passing through dry-run validation first (in development/testing).

**NFR-10** — The system shall handle vault sizes up to 50,000 notes without architectural changes.

### 4.3 Maintainability

**NFR-11** — The rule registry shall be human-readable and editable without running the application.

**NFR-12** — All agent decisions shall be traceable to a specific rule or agent + model run in the audit log.

**NFR-13** — The system shall expose a `/health` endpoint returning structured JSON suitable for monitoring by Home Assistant or similar tools.

## 5. Configuration

All configuration shall live in `~/.librarian/config.yaml`:

```yaml
vault_path: "/Users/ericmedlock/Documents/default obsidian vault"
lm_studio:
  base_url: "http://localhost:1234/v1"
  main_model: "qwen2.5-32b-instruct-q4_k_m"
  embed_model: "nomic-embed-text-v1.5"

api:
  host: "0.0.0.0"
  port: 8080

privacy:
  default_external_tier: 1   # max tier exposed via external API
  sensitive_keywords:         # triggers tier-2 auto-classification
    - therapy
    - medication
    - diagnosis
    - financial
    - password
    - credential

organization:
  autonomous_mode: true       # false = queue changes for approval
  stale_threshold_days: 90
  archive_threshold_days: 180
  rule_generation_batch_size: 25
  rule_confidence_threshold: 0.85
  debounce_seconds: 5

scheduler:
  full_scan_cron: "0 2 * * *"  # 2 AM nightly

notifications:
  pending_actions_threshold: 10
```

## 6. Use Cases

### UC-01: New note saved while writing
User saves a rough note about a conversation with Avery's therapist. The system detects therapy-related keywords, assigns `privacy_tier: 2`, moves the note to `Personal Knowledge Base/Avery/`, adds frontmatter with `topic: Parenting`, `note_type: fleeting`, and indexes it in ChromaDB. No external API access is granted to this note. Rule registry fires on the keyword pattern — no LLM required.

### UC-02: Home assistant asks a question
User's Home Assistant says "What did I learn about parallel computing this semester?" The assistant sends this to `http://librarian.local:8080/v1/chat/completions`. The system embeds the query, retrieves top chunks from `Parallel Computing ITCS 6145.md` and related notes (tier 0), synthesizes a response, and returns it with wikilink citations.

### UC-03: Backup file cleanup
A note named `Capt Disillusion Analysis_backup_20251030_110939.md` is found. Rule `rule_001` (regex: `_backup_\d{8}_\d{6}\.md$`) fires instantly. The file is flagged as a duplicate of its parent. The pending action appears in the dashboard. User clicks approve → file is deleted.

### UC-04: Loose daily notes
18 daily notes are found at vault root during nightly scan. Rule `rule_002` (regex: `^Daily - \d{2}-\d{2}-\d{4}\.md$`) fires for each. All 18 are moved to `Daily Notes/`. Reported in the morning health digest note.

### UC-05: Rule generation from patterns
Over 3 weeks, the Classifier agent has tagged 31 notes containing the word "ITCS" with `topic: Computer Science` and `note_type: reference`. RuleWriter detects the pattern, writes a `frontmatter_pattern` rule: "notes with 'ITCS' in title → topic: Computer Science." Future course notes are tagged at zero LLM cost.

### UC-06: User queries vault from terminal
```bash
librarian query "what do I know about AWS certification prep?"
```
Returns a synthesized answer with source citations from `AWS Practitioner Cert Prep.md` and related notes.

## 7. Out of Scope (v1.0)

- Multi-user or shared vault support
- External cloud sync or backup (user manages their own)
- Web clipping / email ingestion (future roadmap)
- Mobile app (iOS/Android)
- Fine-tuning or local model training
- Obsidian plugin (the system operates at the filesystem level, not inside Obsidian)
- Windows or Linux support (macOS-specific FSEvents and launchd used)
- Canvas file (`.canvas`) organization — text notes only in v1
